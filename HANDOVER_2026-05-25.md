## 핵심 데이터 흐름

```
[견적]   project_estimates + estimate_items
   ├─ items.program_id ───→ programs.id (수동 매핑)
   └─ items.payroll_expense_id ─→ payroll_expenses.id (선택 변환)

[지출]   payroll_expenses
   ├─ expense_type prefix
   │    ├─ 강사료/촬영/기타외주 → 인건비 (isOutsourceType)
   │    └─ 운영비-*/운영인건비   → 운영비 (isOperationType)
   ├─ program_id
   ├─ order_index (수동 정렬)
   ├─ tax_rate_type ('3.3'/'8.8'/'10'/'면세'/'없음')
   └─ subtotal/tax_amount/net_amount (GENERATED 계산)

[지급요청]  프로그램 → PaymentRequestTab
   ├─ 메인 PaymentSummaryCards (제안 견적 vs 실집행 비교)
   ├─ Sub [인건비][운영비] + 행별 [↑][↓][수정][삭제]
   └─ [⬇ 견적에서 가져오기] EstimateImportModal (선택·수정·전송)

[재무요약]  프로젝트 → OverviewTab → FinanceSummaryCard
   ├─ 제안 견적 (proposalTotal)
   ├─ 수입 = income + income_contracts(deposited_at)
   └─ 지출 = expenses + payroll_expenses → 인건비/운영비/대기 분리

[정산]  SettlementPage
   └─ 카드마다 미니 재무 요약 (제안·실집행·잔여) — N+1 회피 일괄 fetch
```

## 미해결 진단 (인수인계 시점)

- **증상**: 프로그램 [지급요청] 행 [🗑 삭제] 시 "삭제 중 오류가 발생했어요" 토스트
- **점검 완료**:
  - `payroll_expenses` 의 `deleted_at`·`order_index` 컬럼 ✅ 존재
  - RLS 정책 ✅ `payroll_expenses_authenticated` (ALL)
  - GRANT ✅ `authenticated` 가 INSERT/SELECT/UPDATE/DELETE 모두 보유
  - `estimate_items`·`project_estimates` RLS 정책 ✅ `_authenticated` (ALL)
- **다음 진단**:
  - 박경수님 브라우저 캐시 가능성 → 시크릿 모드 또는 `Ctrl+Shift+R` 확인
  - 시크릿 모드에서도 "삭제 중 오류가 발생했어요" 라면 → 신규 commit `fbd494e` 배포 안 됨
  - 신규 commit 배포되면 raw 메시지 ("삭제 실패: ...") 가 토스트에 직접 노출 → 진짜 원인 파악 가능

### B. 박경수님 환경의 12건 견적 자동 변환 흔적

- 박경수님 화면에 견적 12건 모두 "집행 대기" 상태 + 외주/급여 페이지에 동일 12건 강사료 prefix
- 박경수님 의도와 다르게 일괄 변환됨 (이전 commit `ee0ec38` 이전 시점)
- 정리 SQL:
```sql
DELETE FROM payroll_expenses
WHERE program_id IS NOT NULL
  AND id IN (
    SELECT payroll_expense_id FROM estimate_items
    WHERE payroll_expense_id IS NOT NULL
  )
  AND payment_status = '대기'
RETURNING id, expense_type, payee_name;
-- 그 후
UPDATE estimate_items SET payroll_expense_id = NULL
WHERE program_id = '<프로그램 UUID>';
```
(박경수님 의도 확인 후 진행 — 정리하면 견적 상태 "미연결" 복원)
